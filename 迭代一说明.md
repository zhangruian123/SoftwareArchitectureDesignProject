# 微型内存文件系统（迭代一）需求规格说明书

## 一、 项目概述

- **截止时间**：6月1日 0 a.m.（暂定）
- **开发目标**：实现一个基于命令行的微型内存文件系统。程序需通过标准输入（Standard Input）读取终端操作指令，在内存中动态构建并维护一颗文件树，并将指令的执行结果输出到标准输出（Standard Output）。

## 二、 领域模型定义

系统中的核心实体分为以下三类：

- **根目录（Root）**：系统的唯一入口，路径固定表示为 `/`。
- **文件（File）**：包含“名称”和“内容大小”（整数类型，模拟字节数）。
- **目录（Directory）**：包含“名称”以及若干子节点（文件或子目录）。目录自身的大小为其内部所有子节点大小的递归相加之和。

## 三、 指令集与系统行为

程序需持续读取指令并执行。如果指令触发“报错/异常”，系统的标准处理方式为：**不输出任何内容，直接忽略该条指令，继续处理下一条。**

| **指令语法**                  | **行为描述**               | **异常与覆盖处理规则**                                       |
| ----------------------------- | -------------------------- | ------------------------------------------------------------ |
| **`MKDIR <绝对路径>`**        | 创建指定名称的目录。       | **父目录不存在**：忽略指令。 **目标已是同名目录**：保持原样（不报错）。 **目标已是文件**：直接销毁原文件，替换为新目录。 |
| **`TOUCH <绝对路径> <大小>`** | 创建指定名称和大小的文件。 | **父目录不存在**：忽略指令。 **目标已是同名文件**：覆盖原文件，更新为新大小。 **目标已是目录**：直接销毁原目录，替换为新文件。 |
| **`LS <绝对路径>`**           | 列出目标路径的信息。       | **目标是目录**：按字母顺序打印直接子节点名称，每行一个。 **目标是文件**：打印该文件名本身。 *(注：测试用例保证LS的目标路径一定存在)* |
| **`INFO <绝对路径>`**         | 查询目标路径的大小。       | **目标是文件**：打印文件大小。 **目标是目录**：打印递归计算后的目录总大小。 *(注：测试用例保证INFO的目标路径一定存在)* |

## 四、 关键约束与特殊规则（必读）

迭代一的判定规则基于“极简模型”，部分逻辑与真实操作系统有较大差异，必须严格按照以下规则实现以通过 OJ 评测。

### 1. 严格字面路径匹配（禁止规范化）

在迭代一中，系统只接受最标准的绝对路径书写方式。以下情况均被视为**非法路径**，包含此类路径的指令必须被**直接忽略**：

- 出现连续斜杠：`//`
- 非根目录但以斜杠结尾：`/usr/`
- 包含特殊路径段：`.` 或 `..`

### 2. 暴力的跨类型覆盖语义

真实系统中文件与目录不能直接互相覆盖，但在本系统中：

- `TOUCH` 可以直接把已存在的目录覆盖成文件。
- `MKDIR` 可以直接把已存在的文件覆盖成目录。

## 五、 输入输出规范示例

**输入格式**：多行字符串，每行一条指令，参数之间以空格分隔。

**输出格式**：对于有返回值的指令（`LS`, `INFO`），每条结果占一行。

**测试用例模拟**：

Plaintext

```
// 标准输入
MKDIR /usr
MKDIR /usr/local
TOUCH /usr/local/test.txt 100
TOUCH /readme.md 50
LS /
INFO /
INFO /usr

// 标准输出
readme.md
usr
150
100
```

## 六、 架构设计建议（为迭代二铺垫）

迭代二将会引入路径规范化（处理 `//` 和 `..`）、文件树递归搜索（`FIND`）以及最复杂的图结构链接（`LINK`）。为了防止后续代码重构，建议在迭代一采用以下设计：

1. **节点多态化**：使用接口或抽象类（如 `Node`）统一定义文件与目录，对外暴露获取名字、类型、大小的通用方法，避免主流程中出现大量类型推断分支。
2. **剥离路径处理逻辑**：将路径的字符串切分、校验逻辑封装到独立的 `Path` 工具类中，确保主业务逻辑只和解析好的节点集合打交道。
3. **天然排序的目录容器**：在目录内部使用有序的 Map 结构（如 Java 的 `TreeMap` 或 C++ 的 `std::map`）来存储子节点，从而自动满足 `LS` 指令的字母序输出要求。
4. **预留防环上下文**：在节点的获取大小（Size）接口中，提前预留一个上下文参数对象（如 `SizeContext`）。迭代一中可暂时不用，但在迭代二引入链接导致文件树变成图时，可直接在上下文中加入已访问节点的 Set 集合来解决死循环问题。
